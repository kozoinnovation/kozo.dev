# AI駆動開発の実践

> Cursor + Claude/GPT で開発速度を10倍にする方法

---

## AIは「コード生成器」ではない

よくある誤解：「AIにコードを書かせる」

私の使い方：「AIに**設計を理解させて**、コードを書かせる」

この違いが、使いこなせる人とそうでない人を分ける。

---

## AI_GUIDE.md という発明

ZeroBookには `AI_GUIDE.md` というファイルがある。

このファイルには以下が書かれている：

- DBスキーマの全体像
- 命名規則
- ディレクトリ構造
- やってはいけないこと
- 既存の実装パターン

AIにコードを書かせる前に、まずこのファイルを読み込ませる。

```
「AI_GUIDE.md を読んで、この設計思想を理解した上で、新しい機能を実装してください」
```

すると、**既存コードと一貫性のある**コードが生成される。

---

## DBスキーマ → 完全なモジュール

最も強力なパターン：

1. DBスキーマを設計する
2. そのスキーマをAIに渡す
3. CRUD API + 管理画面 + フロントエンド を一括生成

```sql
CREATE TABLE menus (
  id UUID PRIMARY KEY,
  name TEXT NOT NULL,
  price INTEGER NOT NULL,
  duration INTEGER NOT NULL,
  is_active BOOLEAN DEFAULT true
);
```

このスキーマを渡すだけで、AIは：

- `/api/menus` のCRUD API
- 管理画面の一覧・詳細・編集フォーム
- フロントエンドのコンポーネント

を全て生成できる。

## なぜこれが可能か

### 1. パターンの一貫性

KozoFrameworkの設計パターンは統一されている。AIはそのパターンを認識し、再現する。

### 2. 型の厳密性

TypeScript + Supabaseの型システムにより、AIが生成したコードの整合性が担保される。

### 3. 人間によるレビュー

AIが生成したコードは必ず人間がレビューする。明らかなミスは修正し、設計判断が必要な部分は自分で書く。

| 工程 | 従来 | AI駆動 |
|------|------|--------|
| 設計 | 2時間 | 2時間（変わらない） |
| CRUD実装 | 4時間 | 30分 |
| UI実装 | 4時間 | 1時間 |
| テスト・修正 | 2時間 | 1時間 |
| **合計** | **12時間** | **4.5時間** |

設計は変わらない。**実装が圧縮される**。

---

## AIに任せないこと

- **設計判断**: 「この機能は必要か？」はAIに聞かない
- **セキュリティの最終確認**: RLSポリシーは必ず人間がレビュー
- **ビジネスロジックの核心**: 予約の競合判定など、間違えると致命的な部分

AIは「作業」を高速化する。「判断」は人間がする。

---

## まとめ

これがAI駆動開発の本質。

コードを書く時間が1/10になれば、その分**設計と思考**に時間を使える。
