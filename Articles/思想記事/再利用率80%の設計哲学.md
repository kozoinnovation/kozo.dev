# 再利用率80%の設計哲学

> 新しいSaaSを作るとき、8割のコードは書かない

---

## 前提：ソロ開発者のリソースは有限

フルタイムのエンジニアチームがいるわけではない。1人で全てをやる。

だからこそ、「毎回ゼロから作る」という選択肢はない。

---

## KozoFrameworkという解

ZeroBookを作った後、私はそのコードを分解した。

**「これは予約システム固有か？　それとも汎用的か？」**

この問いを全ファイルに投げかけた結果、**80%以上**のコードが汎用的だと分かった。

### 汎用的なもの（抽出した）

- 認証・セッション管理
- RLSポリシーのテンプレート
- API エラーハンドリング
- メール送信基盤
- Stripe 連携
- プッシュ通知基盤
- 埋め込み用 Shadow DOM

### ドメイン固有のもの（20%以下）

- 予約ロジック
- 空き状況計算
- メニュー・スタッフ管理

---

## 再利用の実践

新しいSaaSを作るとき、こうする：

1. KozoFrameworkをベースにプロジェクト生成
2. ドメイン固有のテーブル設計
3. AI に DBスキーマを読み込ませ、CRUD + UI を自動生成
4. 微調整

開発時間は従来の **1/5〜1/10** になる。

---

## なぜ80%なのか

100%を目指さない理由がある。

- **100%再利用 = 過度な抽象化** → 逆にメンテナンスが困難に
- **80%再利用 = ちょうど良い抽象化** → ドメイン固有の部分は自由に書ける

「8割は使い回す、2割は自由に書く」

このバランスが、ソロ開発者にとって最も生産的だと分かった。

---

## 帝国構想との接続

複数のSaaSを展開するとき、再利用率80%は**複利**で効いてくる。

```
1プロダクト目：100%の工数
2プロダクト目：20%の工数
3プロダクト目：20%の工数
...
```

5プロダクトを作っても、工数は**1.8倍**程度。これが「SaaS帝国」のスケーラビリティ。

---

## まとめ

> **「再利用できるものは再利用する。できないものだけ書く。」**

シンプルだが、これを徹底するだけで、ソロ開発者でも複数プロダクトを運用できる。
